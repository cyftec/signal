Reactivity Model

How Signal's dependency tracking and automatic updates work.

Global Variable Tracking

Signal uses a global variable-based dependency tracking system instead of explicit dependency graphs. This is a key architectural difference from many other signal libraries.

How It Works

  1. When effect() is called, it sets a module-level _currentSignalEffect variable before executing the function
  2. When a signal's .value getter is called, it checks if _currentSignalEffect exists
  3. If it exists, the effect is added to the signal's internal _effects Set
  4. After effect execution, _currentSignalEffect is set back to null

Implications

  • Effects must access .value during execution to establish dependencies
  • No explicit dependency graph - each signal maintains a Set of registered effects
  • Dependencies are tracked implicitly through the global variable during execution
  • Simpler implementation but less flexible for nested scenarios

Dependency Establishment

An effect or derived signal establishes a dependency on a signal only if the signal's .value getter is accessed during execution.

Important

If a signal's .value is accessed conditionally and the condition is false during initial execution, that signal will not be tracked as a dependency.

Example

// WRONG - count won't be tracked if shouldShow is false initially
const shouldShow = signal(false);
const count = signal(5);
const displayed = derive(() => {
  if (shouldShow.value) {
    return count.value;
  }
  return 0;
});

// CORRECT - Always access the signal
const displayed = derive(() => {
  return shouldShow.value ? count.value : 0;
});

No Automatic Dependency Collection

Unlike some other signal libraries, Signal does not automatically collect dependencies beyond what is accessed during execution.

Guarantees

  • Dependencies are established during the initial execution of an effect or derived signal computation
  • Once a dependency is established, the effect or derived signal will re-run whenever that signal's value changes
  • Dependencies are not automatically re-evaluated on subsequent executions

Non-Guarantees

  • No automatic dependency collection beyond what is accessed during execution
  • No guarantee that conditional dependencies will be tracked if the condition is false during initial execution
  • No guarantee about the internal mechanism of dependency tracking

Synchronous Execution

Signal has no batching or deferred execution. All updates are synchronous and immediate.

Execution Flow

  1. Signal value is set via setter
  2. Signal checks if new value equals old value (short-circuits if equal)
  3. Signal updates internal value using immut()
  4. Signal iterates through its _effects Set
  5. For each effect: if canDisposeNow is true, remove it; otherwise, execute it
  6. Effects may access other signals' .value, triggering their effects recursively

Guarantees

  • Setting a signal's value to a new value triggers all registered effects synchronously
  • Setting a signal's value to the same value does not trigger effects
  • Effects are triggered in the order they were registered to the signal
  • When an effect runs, it may access other signals' values, triggering their effects recursively
  • All propagation is synchronous and immediate

No Batching

Signal has no batching mechanism. Every signal update triggers all dependent effects immediately.

Note

This is different from SolidJS, Preact Signals, and Angular Signals, which all have batching mechanisms.

Implications

  • Multiple signal updates in sequence will trigger effects for each update
  • No manual batch API is provided
  • Each signal update triggers all dependent effects immediately
