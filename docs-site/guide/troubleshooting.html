Troubleshooting

Common issues and how to resolve them when using Signal.

Effect Not Running

Problem: An effect doesn't re-run when a signal changes.

Cause: The signal's .value wasn't accessed during the effect's initial execution, so it wasn't tracked as a dependency.

Solution: Always access the signal's value, even if you don't use it immediately:

// WRONG
const shouldShow = signal(false);
const count = signal(5);
effect(() => {
  if (shouldShow.value) {
    console.log(count.value);
  }
});

// CORRECT
effect(() => {
  if (shouldShow.value) {
    console.log(count.value);
  } else {
    count.value; // Access to track dependency
  }
});

Derived Signal Not Updating

Problem: A derived signal doesn't update when its dependencies change.

Cause: Similar to effects, dependencies are only tracked if accessed during the initial computation.

Solution: Use ternary operators to ensure all dependencies are accessed:

// WRONG
const displayed = derive(() => {
  if (shouldShow.value) {
    return count.value;
  }
  return 0;
});

// CORRECT
const displayed = derive(() => {
  return shouldShow.value ? count.value : 0;
});

Effect Runs Too Many Times

Problem: An effect runs more frequently than expected.

Cause: Setting the same value multiple times triggers effects each time (strict equality check).

Solution: Check if the value actually changed before setting:

// Only set if value is different
if (count.value !== newValue) {
  count.value = newValue;
}

Memory Leak from Effects

Problem: Effects continue running after they're no longer needed.

Cause: Effects weren't disposed when cleanup should have occurred.

Solution: Always dispose effects and derived signals when done:

const count = signal(0);
const doubled = derive(() => count.value * 2);
const eff = effect(() => console.log(count.value));

// Dispose when cleanup is needed
dispose(doubled, eff);

Object Updates Not Working

Problem: Object signal updates don't work as expected with nested properties.

Cause: set() performs shallow merge, not deep merge.

Solution: Replace the entire object for deep updates:

const user = signal({ 
  name: "John", 
  address: { city: "NYC" } 
});

// For deep updates, replace entire object
user.value = { 
  ...user.value, 
  address: { ...user.value.address, city: "LA" } 
};

Trap Methods Not Available

Problem: Expected trap methods are not available on a trap object.

Cause: Trap type is determined at creation time and doesn't change if the signal's value type changes.

Solution: Ensure the signal's value type is stable, or recreate the trap when the type changes:

const value = signal(5);
const trapped = trap(value); // NumberSignalTrap

// If type might change, check and recreate
if (typeof value.value !== "number") {
  // Recreate trap for new type
}

Promise Result Not Updating

Problem: Promise result signal doesn't update after error.

Cause: The promstates API preserves the last successful result on error.

Solution: Always check the error signal before using the result:

const [runPromise, result, error] = promstates(fetchData, null);

effect(() => {
  if (error.value) {
    console.error("Error:", error.value);
  } else {
    console.log("Result:", result.value);
  }
});

Effect Throws Error

Problem: An effect throws an error and stops running.

Cause: Errors in effects propagate immediately and may prevent subsequent runs.

Solution: Wrap effect code in try-catch blocks:

effect(() => {
  try {
    // Your effect code
    updateUI(count.value);
  } catch (error) {
    console.error("Effect error:", error);
  }
});
