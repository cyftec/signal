Code Generation Guidance

API documentation for AI code generation with Signal.

This section provides comprehensive API documentation for generating correct Signal code. For detailed behavioral information, refer to the Behavioral Inventory.

Core Primitives

signal<T>(input: T): SourceSignal<T>

Creates a mutable source signal from any JavaScript value.

signal<T>(input: T): SourceSignal<T>

Returns

  • type: "source-signal" - Runtime type discriminator
  • value: T - Getter/setter for the signal's value
  • For arrays: mutation methods (push, pop, splice, etc.)
  • For objects: set(partial) method for partial updates

derive<T>(valueGetterFn: (oldValue: T | undefined) => T): DerivedSignal<T>

Creates a read-only derived signal computed from other signals.

derive<T>(valueGetterFn: (oldValue: T | undefined) => T): DerivedSignal<T>

Returns

  • type: "derived-signal" - Runtime type discriminator
  • value: T - Getter for current computed value
  • prevValue: T | undefined - Getter for previous computed value
  • dispose(): void - Stops tracking dependencies

effect(fn: () => void): SignalsEffect

Registers a function to run whenever its accessed signals change.

effect(fn: () => void): SignalsEffect

Returns

  • canDisposeNow: boolean - Flag indicating disposal status
  • dispose(): void - Marks the effect for disposal

dispose(...derivedSignalsOrEffects): void

Disposes multiple derived signals and/or effects at once.

dispose(...derivedSignalsOrEffects: (DerivedSignal<any> | SignalsEffect)[]): void

API Layer

compute<F, Args>(computerFn: F, ...restArgs): DerivedSignal<ReturnType<F>>

Creates a derived signal from a function with signalified arguments.

compute<F extends (...args: any[]) => any>(computerFn: F, ...restArgs: MaybeSignalValues<Parameters<F>>): DerivedSignal<ReturnType<F>>

receive<T>(receiver: SourceSignal<T>, ...transmittors): SignalsEffect[]

Connects multiple transmitter signals to a receiver signal.

receive<T>(receiver: SourceSignal<T>, ...transmittors: Signal<T>[]): SignalsEffect[]

transmit<T>(transmittor: Signal<T>, ...receivers): SignalsEffect

Broadcasts changes from one transmitter signal to multiple receiver signals.

transmit<T>(transmittor: Signal<T>, ...receivers: SourceSignal<T>[]): SignalsEffect

promstates<R, Args, I>(promiseFn, initialValue?, ultimately?): readonly [runPromise, result, error, isRunning]

Creates promise state signals for async operations.

promstates<R, Args extends Array<any>, I>(promiseFn: (...args: Args) => Promise<R>, initialValue?: I, ultimately?: () => void): readonly [(...args: Args) => Promise<void>, DerivedSignal<R | I | undefined>, DerivedSignal<Error | undefined>, DerivedSignal<boolean>]

tmpl(strings, ...tlExpressions): DerivedSignal<string>

Tagged template function for string interpolation with signals.

tmpl(strings: TemplateStringsArray, ...tlExpressions: StringSignalDeriverTemplateExpressions): DerivedSignal<string>

trap<T>(input: MaybeSignalValue<T>): SignalTrap<T>

Creates a type-specific trap object with convenient derived signal methods.

trap<T>(input: MaybeSignalValue<T>): SignalTrap<T>

op<T>(input: MaybeSignalValue<T> | (() => T)): Operation<T>

Creates an operation object for composing logical and mathematical operations on signals.

op<T>(input: MaybeSignalValue<T> | (() => T)): Operation<T>

Type Checkers

valueIsSourceSignal(input): boolean

Checks if a value is a source signal.

valueIsSourceSignal(input: MaybeSignal<any>): boolean

valueIsDerivedSignal(input): boolean

Checks if a value is a derived signal.

valueIsDerivedSignal(input: MaybeSignal<any>): boolean

valueIsSignal(input): boolean

Checks if a value is any signal (source or derived).

valueIsSignal(input: MaybeSignal<any>): boolean

value<T>(input: MaybeSignalValue<T>): T

Extracts the plain value from a signal, non-signal, or plain value.

value<T>(input: MaybeSignalValue<T>): T

Code Generation Best Practices

  • Always access signals - Ensure effects and derived signals always access .value even in conditional branches
  • Use ternary operators - Prefer ternary over if-else for derived signals to ensure dependency tracking
  • Dispose when done - Always dispose effects and derived signals when no longer needed
  • Check error first - For promstates, always check error before using result
  • Avoid type changes - Don't change signal value types after creating traps or operations
  • Handle errors - Wrap effect code in try-catch blocks
