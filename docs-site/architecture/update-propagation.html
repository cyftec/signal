Update Propagation

Signal's scheduling strategy and execution flow.

Synchronous Execution

Signal has no batching or deferred execution. All updates are synchronous and immediate.

  • When a signal's value is set, all registered effects run immediately
  • Effects run in the order they were registered
  • No scheduling queue or microtask timing

Note

This is different from SolidJS, Preact Signals, and Angular Signals, which all have batching mechanisms.

Effect Execution Flow

When a signal value is set, the following execution flow occurs:

  1. Signal value is set via setter
  2. Signal checks if new value equals old value (short-circuits if equal)
  3. Signal updates internal value using immut()
  4. Signal iterates through its _effects Set
  5. For each effect: if canDisposeNow is true, remove it; otherwise, execute it
  6. Effects may access other signals' .value, triggering their effects recursively

Example

const count = signal(0);
const doubled = derive(() => count.value * 2);

effect(() => {
  console.log("Count:", count.value);
  console.log("Doubled:", doubled.value);
});

count.value = 1;
// 1. Signal value is set to 1
// 2. Signal checks if 1 !== 0 (true)
// 3. Signal updates internal value
// 4. Signal iterates through effects
// 5. Effect executes
// 6. Effect accesses doubled.value
// 7. This triggers doubled's internal effect
// 8. All happens synchronously

No Batching Mechanism

Signal has no batching mechanism. Every signal update triggers all dependent effects immediately.

Implications

  • All updates are synchronous and immediate
  • No automatic batching of multiple signal updates
  • No manual batch API provided
  • Each signal update triggers all dependent effects immediately

Example

const count = signal(0);
let runCount = 0;

effect(() => {
  runCount++;
  count.value;
});

count.value = 1; // runCount becomes 2 (immediate)
count.value = 2; // runCount becomes 3 (immediate)
count.value = 3; // runCount becomes 4 (immediate)
// Each update triggers the effect immediately

Effect Execution Order

Effects are executed in the order they were registered to the signal.

const count = signal(0);
const order: number[] = [];

effect(() => {
  order.push(1);
  count.value;
});

effect(() => {
  order.push(2);
  count.value;
});

count.value = 1;
console.log(order); // [1, 2] - effects run in registration order

Recursive Effect Triggering

Effects can trigger other effects recursively by accessing other signals' values during execution.

const count = signal(0);
const doubled = derive(() => count.value * 2);

effect(() => {
  console.log("Count:", count.value);
  console.log("Doubled:", doubled.value);
});

count.value = 1;
// Effect runs
// Accessing doubled.value triggers doubled's internal effect
// All happens synchronously in the same call stack

No Scheduling

Signal has no microtask scheduling or deferred execution.

  • All propagation is synchronous
  • Different from most frameworks that use scheduling for performance
  • No queue or deferred execution mechanism

Cleanup During Execution

Disposed effects are removed during the next signal update, not immediately.

Important

Lazy cleanup strategy: disposed effects might run one more time after disposal if a signal update occurs before cleanup.

Example

const count = signal(0);
let runCount = 0;

const eff = effect(() => {
  runCount++;
  count.value;
});

eff.dispose(); // Marks for disposal
// Effect still registered until next signal update
count.value = 1; // Effect is removed here without running
// runCount remains 1
