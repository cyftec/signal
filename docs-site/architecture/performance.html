Performance

Signal's performance characteristics and considerations.

Performance Guarantees

Signal provides minimal performance guarantees:

  • No guarantee about the maximum depth of recursive effect triggering
  • No guarantee about performance characteristics of effect execution
  • No guarantee about the order of effect execution across multiple signals when a single update triggers effects on multiple signals

Important

Performance may vary based on the number of signals, effects, and complexity of computations.

Synchronous Execution Overhead

Since all updates are synchronous, there is no deferred execution overhead.

Benefits

  • Predictable execution timing
  • No microtask queue overhead
  • Immediate feedback on state changes

Considerations

  • Multiple rapid updates may cause many effect executions
  • No automatic batching to coalesce updates
  • Deep effect chains may cause stack depth issues

Memory Overhead

Signal has minimal memory overhead due to its design:

Per-Signal Overhead

  • Source signal: value + Set of effects
  • Derived signal: source signal + effect (internal)
  • Effect: function + disposal flag

No Explicit Graph

  • No dependency graph data structure
  • Each signal maintains only its direct effects
  • Reduced memory footprint compared to graph-based approaches

Immutable Value Storage

Signal values are stored immutably using @cyftech/immutjs.

Benefits

  • Prevents accidental mutations
  • Enables easy change detection via strict equality
  • Consistent behavior across data types

Considerations

  • Immutable copies have memory overhead
  • Large objects/arrays may be expensive to copy
  • Frequent updates to large structures may impact performance

Effect Execution Cost

Effect execution cost depends on the complexity of the effect function.

Factors

  • Number of signals accessed
  • Complexity of computations in the effect
  • Number of nested effects triggered
  • DOM updates or other side effects

Optimization Tips

  • Keep effects focused on single responsibilities
  • Move complex computations to derived signals
  • Dispose effects when no longer needed
  • Avoid deep nesting of effects

Derived Signal Computation Cost

Derived signals recompute whenever dependencies change.

Factors

  • Complexity of the value getter function
  • Number of dependencies
  • Frequency of dependency changes

Optimization Tips

  • Keep derived signal computations simple
  • Use memoization for expensive computations
  • Dispose derived signals when no longer needed
  • Consider splitting complex derived signals

Disposal Performance

Disposal uses a lazy cleanup strategy.

Benefits

  • Disposal is O(1) - just sets a flag
  • Cleanup happens during normal signal updates
  • No separate cleanup pass needed

Considerations

  • Disposed effects may run one more time
  • Cleanup only happens on next signal update
  • Signals that never update won't clean up disposed effects

Comparison with Batched Approaches

Batched Libraries (SolidJS, Preact, Angular)

  • Pros: Coalesce multiple updates, reduce effect executions
  • Cons: Deferred execution, microtask overhead, more complex implementation

Signal (No Batching)

  • Pros: Immediate execution, simpler implementation, predictable timing
  • Cons: More effect executions, no automatic coalescing

Performance Best Practices

  • Dispose when done - Always dispose effects and derived signals when no longer needed
  • Avoid unnecessary updates - Check if values actually changed before setting
  • Keep effects simple - Move complex logic to derived signals
  • Avoid deep nesting - Deep effect chains may cause performance issues
  • Use derived signals for computations - Keep effects focused on side effects
