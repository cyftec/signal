Common Mistakes

Critical gotchas and pitfalls when using Signal.

1. Conditional Dependency Tracking

Critical Gotcha

Dependency tracking only happens for signals accessed during execution. If a signal is accessed conditionally and the condition is false on first run, it won't be tracked.

Wrong

const shouldShow = signal(false);
const count = signal(5);
const displayed = derive(() => {
  if (shouldShow.value) {
    return count.value;
  }
  return 0;
});
// Changing count.value won't trigger recomputation!

Correct

const displayed = derive(() => {
  return shouldShow.value ? count.value : 0;
});

2. Effect Conditional Dependency

Same issue applies to effects:

Wrong

const shouldLog = signal(false);
const count = signal(0);
effect(() => {
  if (shouldLog.value) {
    console.log(count.value);
  }
});
// Changing count.value won't trigger the effect!

Correct

effect(() => {
  if (shouldLog.value) {
    console.log(count.value);
  } else {
    count.value; // Access to track dependency
  }
});

3. prevValue Misunderstanding

Critical Gotcha

The prevValue parameter is the previous RETURN value, not the previous dependency values.

const count = signal(5);
const doubled = derive((prev) => {
  const current = count.value * 2;
  // prev is the previous doubled value (10), not the previous count value
  return current;
});

4. Disposal Timing

Critical Gotcha

Disposal is lazy - effects are removed on the next signal update, not immediately. Disposed effects might run one more time after disposal.

const count = signal(0);
const eff = effect(() => console.log(count.value));

eff.dispose();
count.value = 1; // Effect is removed here without running
// But if no signal update happens, effect remains registered

5. Trap Type Changes

Critical Gotcha

The trap type is determined at creation time and doesn't change if the signal's value type changes.

const value = signal(5);
const trapped = trap(value); // NumberSignalTrap
value.value = "hello"; // Type change not reflected
// trapped still has number methods, not string methods

6. Operation Type Changes

Same issue applies to operations:

const value = signal(5);
const operation = op(value); // NumberOperation
value.value = "hello"; // Type change not reflected
// operation still has number methods, not string methods

7. Object Deep Merge

Critical Gotcha

Object set() performs shallow merge, not deep merge.

const user = signal({ 
  name: "John", 
  address: { city: "NYC" } 
});

user.set({ address: { city: "LA" } });
// user.value is { name: "John", address: { city: "LA" } }
// Shallow merge, not deep merge

8. Promise Error Handling

Critical Gotcha

Always check error first while using promstates. The result is preserved on error.

const [runPromise, result, error] = promstates(fetchData, null);

effect(() => {
  if (error.value) {
    console.error("Error:", error.value);
  } else {
    console.log("Result:", result.value);
  }
});

9. Same Value Updates

Critical Gotcha

Setting the same value twice won't trigger effects (strict equality check).

const count = signal(0);
let runCount = 0;

effect(() => {
  runCount++;
  count.value;
});

count.value = 0; // No change, effect doesn't run
console.log(runCount); // Still 1

10. No Automatic Disposal

Critical Gotcha

No automatic disposal based on scope or lifecycle. Manual disposal required.

// WRONG - Effect never gets disposed
function createComponent() {
  const count = signal(0);
  effect(() => console.log(count.value));
  // Effect leaks when component is destroyed
}

// CORRECT - Manual disposal
function createComponent() {
  const count = signal(0);
  const eff = effect(() => console.log(count.value));
  
  return () => {
    eff.dispose();
  };
}

11. Error in Effect

Critical Gotcha

If an effect throws an error during execution, the error propagates immediately and may prevent subsequent runs.

// Wrap in try-catch
effect(() => {
  try {
    // Your effect code
    updateUI(count.value);
  } catch (error) {
    console.error("Effect error:", error);
  }
});
