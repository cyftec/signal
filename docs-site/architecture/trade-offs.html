Design Trade-offs

Areas where Signal differs from common signal libraries.

1. Global Variable Dependency Tracking

Difference

Most libraries use explicit dependency graphs or stacks. Signal uses a single module-level variable.

Trade-off

  • Simpler - Less complex implementation
  • Less flexible - Not suitable for nested scenarios

2. No Batching

Difference

SolidJS, Preact Signals, Angular Signals all have batching. Signal has no batching mechanism.

Trade-off

  • Predictable - Immediate, synchronous execution
  • More executions - Every update triggers effects immediately

3. Derived Signal Implementation

Difference

Most libraries have dedicated derived signal nodes. Signal wraps a source signal + effect.

Trade-off

  • Consistent - Reuses existing primitives
  • Indirect - More layers of abstraction

4. Array Mutation Methods

Difference

Most signal libraries treat arrays as immutable. Signal provides mutating methods (push, pop, etc.).

Trade-off

  • Familiar API - Feels like standard array methods
  • Internally immutable - Methods create new arrays internally

5. Custom Array Methods

Difference

Signal includes remove() method (inverse of filter), not standard in JavaScript or other signal libraries.

Trade-off

  • Convenient - Useful for common patterns
  • Non-standard - Not available in other libraries

6. No Scheduling

Difference

No microtask scheduling or deferred execution. All propagation is synchronous.

Trade-off

  • Simple - No scheduling complexity
  • Different - Unlike most frameworks that use scheduling for performance

7. Type Discrimination

Difference

Uses type property on all signal objects for runtime type checking.

Trade-off

  • Flexible - Enables runtime type checking for complex scenarios
  • Uncommon - Not common in other signal libraries

8. Non-Signal Objects

Difference

Explicit runtime type wrapper for plain values. Used for type discrimination in complex type scenarios.

Trade-off

  • Powerful - Enables advanced type patterns
  • Verbose - Requires explicit wrapping

9. Effect Registration Condition

Difference

Effects only register to signals if .value is accessed during execution. Different from automatic dependency collection.

Trade-off

  • Explicit - Clear when dependencies are tracked
  • Error-prone - Easy to miss dependencies with conditional logic

10. Disposal Cleanup Timing

Difference

Disposed effects are not removed immediately. They are removed on the next signal update.

Trade-off

  • Lazy - Cleanup happens during normal updates
  • Delayed - Effects may run one more time after disposal

11. Array Signal Implementation

Difference

Array mutation methods create new arrays internally but the API feels like standard array mutation.

Trade-off

  • Familiar - Standard array method names
  • Surprising - Internally immutable but feels mutable

12. Derived Signal Previous Value

Difference

Derived signals expose prevValue getter. This is uncommon in other signal libraries.

Trade-off

  • Useful - Helpful for diffing or change detection
  • Extra - Additional state to maintain

13. Object Signal Partial Updates

Difference

Object signals have set(partial) method for partial updates. Different from full replacement.

Trade-off

  • Convenient - Easy partial updates
  • Shallow - Only shallow merge, not deep merge

14. No Automatic Disposal

Difference

No automatic disposal based on scope or lifecycle. Manual disposal required.

Trade-off

  • Explicit - Clear lifecycle management
  • Manual - Requires developer discipline

Summary

Signal makes intentional design decisions that differ from common signal libraries:

  • Simplicity over complexity - Global variable tracking, no batching, no scheduling
  • Explicit over implicit - Manual disposal, explicit dependency access
  • Familiarity over purity - Array mutation methods, partial object updates
  • Flexibility over constraints - Type discrimination, non-signal objects

These trade-offs make Signal simpler to understand and implement, while requiring more explicit patterns from developers.
